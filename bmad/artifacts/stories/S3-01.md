# Story S3-01 – Migrate `ClassApp::query()` and `findOne()` to modern MongoDB driver

**Epic:** ClassApp CRUD Migration  
**Sprint:** 03  
**Points:** 3  
**Priority:** Must  
**Traces to:** Phase 2 Strategy (PHASE2_STRATEGY.md), MongoCompat (MONGOCOMPAT.md)

---

## User Story

As a **back-end developer**, I want `ClassApp::query()` and `findOne()` migrated from the legacy MongoClient extension to the modern `MongoDB\Client` library,  
so that the core data access layer uses the current official MongoDB driver with proper type safety, filtering, and MongoCompat integration.

---

## Context

`ClassApp` is the ORM entry point for all MongoDB queries in Idae. Currently:
- `getMongoClient()` returns a `MongoDB\Client` instance (S2-05 completed)
- `query()` and `findOne()` still use legacy driver syntax (e.g., `MongoCursor`, deprecated patterns)
- Tests run against the Docker test sidecar (`mongo-test` on port 27018)

This story:
1. **Refactors query() signature** to use `find()` with filters, options, typeMap
2. **Refactors findOne()** to use `findOne()` with MongoCompat filter conversion
3. **Adds unit tests** in `Unit/ClassAppTest.php` covering read operations
4. **Adds strict type hints** and English PHPDoc
5. **Validates against MongoCompat** for all filter/result transformations

## Execution Plan

### Phase 1: Understand Current Implementation

1. **Review** `ClassApp::query()` and `findOne()` in `idae/web/appclasses/ClassApp.php`:
   - Signature (parameters, return type)
   - Filter transformation logic
   - Cursor/result handling
   - Call sites in services (e.g., `json_data.php`)

2. **Identify filter patterns** used in the codebase:
   - Simple filters: `['idproduit' => 123]`
   - ObjectId filters: `['_id' => '507f1f77bcf86cd799439011']` (string repr)
   - Regex filters: `['name' => new MongoRegex('/pattern/')]`
   - Complex filters: `{'$and', '$or', '$ne', etc.}` (MongoDB query operators)

### Phase 2: Refactor query() Method

**Current pattern (legacy):**
```php
function query($collection, $filter = []) {
  $cursor = $this->db->{$collection}->find($filter);
  // Return cursor expecting fetchRow() calls
}
```

**New pattern (modern):**
```php
declare(strict_types=1);

/**
 * Execute a query against a collection.
 * 
 * @param string $collection Collection name
 * @param array<string, mixed> $filter Query filter (MongoCompat-compatible)
 * @param array<string, mixed> $options Query options (projection, sort, limit, etc.)
 * @return \MongoDB\Driver\Cursor Cursor object
 */
public function query(string $collection, array $filter = [], array $options = []): \MongoDB\Driver\Cursor {
    // Convert LESS-style filters to MongoCompat format
    $filter = $this->normalizeFilter($filter);
    
    // Execute find()
    $collection_obj = $this->db->selectCollection($collection);
    return $collection_obj->find($filter, $options);
}
```

**Helper: normalizeFilter()**
```php
private function normalizeFilter(array $filter): array {
    // Process each filter key:
    // - _id: apply MongoCompat::toObjectId()
    // - Regex patterns: apply MongoCompat::toRegex()
    // - Dates: apply MongoCompat::toDate()
    // - Leave scalar values as-is
}
```

### Phase 3: Refactor findOne() Method

```php
/**
 * Find a single document.
 * 
 * @param string $collection Collection name
 * @param array<string, mixed> $filter Query filter
 * @param array<string, mixed> $options Query options
 * @return array<string, mixed>|null Document as array or null
 */
public function findOne(string $collection, array $filter = [], array $options = []): ?array {
    $filter = $this->normalizeFilter($filter);
    $collection_obj = $this->db->selectCollection($collection);
    return $collection_obj->findOne($filter, $options);
}
```

### Phase 4: Create Unit Tests

## Blockers & Risks
- **Call site compatibility:** Some code may expect `MongoCursor` methods (e.g., `fetchRow()`) — validate compatibility or add shim.
- **Cursor lifetime:** MongoDB\Client cursors are fully-evaluated in memory if used with typeMap — ensure large result sets don't cause OOM.
- **Filter edge cases:** Complex operators (`$in`, `$regex`, `$where`) need thorough testing.
