# Story S2-03 ‚Äì Convert all LESS mixins to SCSS `@mixin` / `@include`

**Epic:** SCSS Migration Complete  
**Sprint:** 02  
**Points:** 2  
**Priority:** Must  
**Traces to:** SCSS modernization, mixin pattern standardization

---

## User Story

As a **front-end developer**, I want all LESS mixin definitions and calls converted to SCSS `@mixin` and `@include` syntax,  
so that the stylesheet architecture uses modern, standard SCSS patterns and avoids legacy LESS class-as-mixin anti-patterns.

---

## Context

LESS uses `.classname() { ... }` as mixin definitions and `.classname` as calls. SCSS uses `@mixin classname { ... }` and `@include classname;`. This story:

1. **Identifies all mixin definitions** in LESS files (e.g., `.padding_more`, `.rgba_link` in converted SCSS)
2. **Converts definitions** to SCSS `@mixin` syntax
3. **Converts calls** from LESS `.mixin` to SCSS `@include mixin;`
4. **Validates compilation** and tests mixin application

---

## Execution Plan

### Step 1: Identify LESS-style Mixins in Converted SCSS

Common LESS mixins in legacy Idae code (from conversion analysis):
- `.padding_more` ‚Äî adds padding
- `.rgba_link` ‚Äî applies RGBA color/styling to links
- `.animated`, `.slideInRight`, `.speed` ‚Äî animation classes (may need to remain as classes, not mixins)
- `.boxshadow` ‚Äî applies box-shadow
- `.preserve3d` ‚Äî applies CSS 3D perspective (in swiper/mixins)
- `.encoded-svg-background()` ‚Äî applies SVG background encoding

### Step 2: Audit Files for Mixin Definitions & Calls

Files likely to contain mixins:
- `_app_fabric.scss` ‚Äî foundation mixin library
- `_style.scss` ‚Äî style definitions
- `vendor/swiper/_mixins.scss` ‚Äî vendor mixin library
- `main.scss` ‚Äî calls to `@include padding_more` (line 103)

### Step 3: Convert Mixin Definitions

**Pattern:**
- LESS: `.mixin_name() { property: value; }`
- SCSS: `@mixin mixin_name { property: value; }`

If mixins accept parameters:
- LESS: `.mixin_name(@param1, @param2) { property: @param1; }`
- SCSS: `@mixin mixin_name($param1, $param2) { property: $param1; }`

### Step 4: Convert Mixin Calls

**Pattern:**
- LESS: `.selector { .mixin_name; }`
- SCSS: `.selector { @include mixin_name; }`

With parameters:
- LESS: `.selector { .mixin_name(10px, 5px); }`
- SCSS: `.selector { @include mixin_name(10px, 5px); }`

### Step 5: Compile & Validate

1. Run `cd idae/web/app_node && npm run build:css`
2. Verify no SASS errors
3. Inspect generated `appcss/dist/main.css` for mixin output
4. Boot app and visually test pages using mixin-dependent styles

### Step 6: Regression Testing

- [ ] Load UI pages with table/form inputs (uses padding_more)
- [ ] Load pages with links (rgba_link styling)
- [ ] Load pages with animations (if .animated classes are mixin-dependent)
- [ ] Verify no CSS visual regressions

## Blockers & Risks
- Some "mixins" may actually be utility classes (e.g., `.animated` in animate.css) ‚Äî convert only true mixin definitions, not utility classes.
- Parameter syntax (`@param` ‚Üí `$param`) must be correct or SCSS compilation will fail.
- Deep mixin dependencies (one mixin calling another) require careful validation.
