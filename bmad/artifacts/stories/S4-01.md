# Story S4-01 – Extract ClassAppAgg.php

**Epic:** ClassApp CRUD Migration
**Sprint:** 04
**Points:** 3
**Priority:** Must
**Traces to:** PHASE2_STRATEGY.md, sprint-04.md

---

## User Story

As a back-end developer, extract the aggregation/distinct helper methods from
`ClassApp` into a dedicated `ClassAppAgg` helper class so they can be tested,
maintained, and modernized independently.

---

## Acceptance Criteria

- `ClassAppAgg::distinct()`, `ClassAppAgg::distinct_rs()`, `ClassAppAgg::distinct_all()` extracted as static methods.
- Methods accept `\App $app` as first parameter (same pattern as `ClassAppFk`).
- `declare(strict_types=1)` + type hints where possible.
- English PHPDoc on all public methods.
- `MongoCompat::convertFilter()` applied to incoming `$vars`.
- No direct legacy Mongo types (`MongoId`, `MongoRegex`, etc.) in the new class.
- Unit tests in `ClassAppAggTest.php` with at least one passing test per method.
- PHPStan level 6 — 0 violations on new file.

---

## Implementation Notes

**Date:** 2026-03-11
**Files changed:**
- `idae/web/appclasses/appcommon/ClassAppAgg.php` — new static helper class
- `idae/web/tests/Unit/ClassAppAggTest.php` — unit tests

**Notable decisions:**
- Static helper pattern matching `ClassAppFk` (no inheritance, accepts `\App $app`).
- `distinct_rs()` kept structurally intact; only `$this` replaced with `$app`, `MongoCompat::convertFilter()` applied to filter.
- Original methods in `ClassApp` kept as delegating wrappers to preserve call-site compatibility.

**Known limitations:**
- `distinct_rs()` uses `$rs_basedist->count()` via `MongodbCursorWrapper` — still legacy pattern; full cursor migration deferred to Phase 4.
