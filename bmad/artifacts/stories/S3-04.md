# Story S3-04 – Refactor ClassApp CRUD helpers: insert/update/remove

**Epic:** ClassApp CRUD Migration
**Sprint:** 03
**Points:** 3
**Priority:** Must
**Traces to:** PHASE2_STRATEGY.md, MONGOCOMPAT.md

---

## User Story

As a back-end developer, refactor the lower-level CRUD helper methods (insert, update, remove) used by ClassApp so they use the modern MongoDB driver, accept MongoCompat-converted inputs, and preserve legacy return shapes and side effects.

---

## Context

- These helpers are called across service endpoints and templates. They must be backward-compatible at the call-site level.
- Must reuse MongoCompat utilities for ObjectId, Regex, Date conversion.
- Preference for small, non-breaking adapter layer if full signature changes are disruptive.

---

## Acceptance Criteria

- Insert/update/remove use `insertOne`, `updateOne`, `deleteOne` with appropriate options.
- Helper methods return the same logical IDs and arrays as legacy helpers.
- No direct usage of legacy Mongo types (MongoId, MongoRegex, MongoDate) remains in helper code.
- Unit tests added for each helper verifying behavior under normal and edge cases.
- PHPStan reports no new issues on touched files.

---

## Execution Plan (high level)

1. Locate the helper functions in `appclasses/` and `appclasses/appcommon/` (search for `insert`, `update`, `remove` helper usage).
2. Extract helpers into `appclasses/appcommon/ClassAppCrud.php` (class or trait) exposing `insertOne`, `updateOne`, `removeOne` equivalents.
3. Update `ClassApp` to delegate to the new helper methods to preserve public surface.
4. Add unit tests in `idae/web/tests/Unit/ClassAppCrudTest.php`.
5. Run PHPStan and PHPUnit (unit). Fix regressions.
6. Commit changes, update `bmad/status.yaml` and `bmad/dashboard.md`.

---

## Notes

- Avoid widespread call-site changes; prefer adapter methods where necessary.
- Coordinate any API surface change with the team before broad replacements.
