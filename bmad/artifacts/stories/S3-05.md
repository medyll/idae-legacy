# Story S3-05 – Unit tests for ClassApp CRUD methods

**Epic:** ClassApp CRUD Migration
**Sprint:** 03
**Points:** 3
**Priority:** High
**Traces to:** S3-03, S3-04, idae/web/PHASE2_STRATEGY.md

---

## User Story

As a QA/Developer, provide unit tests covering `ClassApp` CRUD methods (create_update, insert, update, remove) and FK helpers to ensure behavior parity and prevent regressions during refactors.

---

## Context

- Tests should target `idae/web/tests/Unit/` using the existing TestCase bootstrap.
- Use MONGO_TEST_DSN or a test Mongo instance configured via environment variables.
- Favor isolated unit tests (mock or in-memory where practical) and small integration tests for driver behavior.

---

## Acceptance Criteria

- Unit tests exist for `create_update()` upsert paths, `insert`, `update`, `remove` helpers, and FK resolution helpers.
- Tests assert legacy return shapes (field-named suffixes) and expected exceptions on invalid input.
- CI (local PHPUnit) passes unit suite for touched files.

---

## Execution Plan

1. Add `idae/web/tests/Unit/ClassAppCrudTest.php` with focused test cases.
2. Add fixture documents in `tests/fixtures/` if needed.
3. Run `vendor/bin/phpunit --testsuite=Unit` and fix any regressions.
4. Commit tests and update `bmad/status.yaml` and dashboard.

---

## Notes

- Keep tests deterministic; reset test collections before each test.
- If a test requires integration with the modern driver, mark it as `@integration` and keep it in Integration suite.
